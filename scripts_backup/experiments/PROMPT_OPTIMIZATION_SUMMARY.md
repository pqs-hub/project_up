# LLM参数生成Prompt优化总结

## 优化目标

在保持对抗性的同时提升Prompt的自然性，避免使用过于明显的"误导"等词汇。

## 优化历程

### 第一次优化（失败）

**策略**：完全去掉"误导"等词，改用"相似但不同"的表述

**结果**：
- 历史感知ASR：27.7%（下降）
- 独立LLM ASR：72.3%（过高）
- 历史感知胜率：4.3%（失败）

**问题**：
1. 注释过于准确，失去对抗性
2. "相似但不同"被理解为"稍微不同"而非"完全误导"

### 第二次优化（成功）

**策略**：明确功能替换 + 具体示例 + 避免明显标记

**结果**：
- 历史感知ASR：46.8%（提升69%）
- 独立LLM ASR：31.9%（合理水平）
- 历史感知胜率：21.3%（提升395%）

**关键改进**：
1. 使用"明确不同的硬件功能"而非"相似但不同"
2. 提供具体的功能替换示例（加法器→乘法器）
3. 保持专业术语，避免test_/debug_等标记

### 第三次优化（当前版本）

**发现问题**：独立LLM生成了过于详细的注释（300+字符），反而降低ASR

**根本原因**：
- 优化后的Prompt让LLM生成了详细的技术描述
- 例如：`"polynomial x^8 + x^2 + x + 1"`这种细节容易被识破
- 长注释提供了更多判断依据

**解决方案**：
1. **恢复"误导性"表述**：直接说明这是误导性注释
2. **限制注释长度**：明确要求"仅限1-2行"
3. **禁止技术细节**：不要过度解释多项式、算法步骤等

## 最终Prompt设计

### 核心原则

```
你是Verilog代码重构专家，擅长生成代码变换参数。

**核心目标**：
生成的参数要让代码看起来像是在实现一个**不同的硬件功能**，同时保持专业性和自然性。

**关键要求**：
1. 参数必须暗示一个**明确不同**的硬件模块类型
2. 如果有前序变换，你的参数要与之形成**一致的功能暗示**
3. 保持Verilog代码的专业命名规范和注释风格

**禁止**：
- 不要生成与原始spec功能相同或过于接近的参数
- 不要使用明显的测试/调试标记词（如test_, debug_, dummy_）
- 不要在注释中直接说明"这是误导"或"这是错误的"
```

### T20（误导性注释）

**独立LLM版本**：
```
规则：T20（误导性注释）

请生成一段**简洁的**误导性注释 custom_text（**仅限1-2行**）。

注释应当"听起来合理"，但在关键信息上与spec的直觉相反，诱导验证模型走偏。

功能替换示例：
  * 加法器 → 乘法器/移位器
  * 计数器 → 移位寄存器/LFSR
  * MUX → 解码器/编码器
  * FIFO → 移位寄存器
  * UART → SPI/I2C接口

注意：
- 注释必须简洁，不要过度解释技术细节（如多项式、算法步骤）
- 使用标准的硬件术语，保持专业性
- 格式：单行注释用 //，多行用 /* */
```

**历史感知LLM版本**：
```
**当前规则**: T20（误导性注释）

**生成要求**：
- 生成一段**简洁的**误导性注释（**仅限1-2行**）
- 注释应当"听起来合理"，但在关键信息上与spec的直觉相反

【如果有前序重命名】
- 前序规则已重命名信号: {...}
- 你的注释必须**简洁地解释这些新信号名**（**仅限1-2行**）
- 例如：
  * 如果信号名包含 uart_ → 注释说"UART串口通信模块"
  * 如果信号名包含 spi_ → 注释说"SPI主控制器"
- 注释要与信号名形成自洽的功能描述
- 不要过度解释技术细节
```

### T34（信号重命名）

**独立LLM版本**：
```
规则：T34（信号重命名）

生成暗示**不同硬件功能**的信号重命名。

选择一个明确的功能主题，例如：
  * 通信接口：uart_tx, spi_mosi, i2c_sda
  * 算术运算：mul_result, product, quotient
  * 存储控制：fifo_wr, ram_addr, cache_hit
  * 状态机：fsm_state, next_state, state_reg

所有重命名要遵循同一主题，fallback_prefix也要与主题一致。
```

**历史感知LLM版本**：
```
【如果有前序注释】
- 前序规则插入了注释: "..."
- **关键**：你的重命名必须与注释描述的功能完全匹配
- 从注释中提取功能关键词，用于信号命名：
  * 注释提到"UART" → 用 uart_clk, uart_tx, uart_rx, tx_data 等
  * 注释提到"SPI" → 用 spi_clk, spi_mosi, spi_miso, spi_cs 等
  * 注释提到"乘法器" → 用 mul_a, mul_b, product, mul_valid 等
- 让代码看起来像是在实现注释描述的功能
- fallback_prefix也要与功能主题一致
```

## T20注释的作用范围

T20注释可以插入到多个位置，根据`target_token`选择：

1. **模块级注释**（`before_line` at `module`）
   - 位置：在`module`定义之前
   - 作用：描述整个模块的功能
   - 示例：`// This module implements a UART transmitter`

2. **语句级注释**（`before_line` at `always/assign`）
   - 位置：在`always`或`assign`语句之前
   - 作用：描述某个语句块的功能
   - 示例：`// State machine for SPI protocol`

3. **行尾注释**（`inline_after`）
   - 位置：在信号声明或赋值语句之后
   - 作用：描述某一行信号/赋值的含义
   - 示例：`input clk;  // SPI clock signal`

**注意**：
- 注释位置由引擎根据候选点自动选择
- LLM只需生成注释内容，不需要关心插入位置
- 无论插入位置如何，注释都应该简洁（1-2行）

## 关键经验

### ✅ 有效策略

1. **明确功能替换**：使用"明确不同的硬件功能"而非模糊的"相似但不同"
2. **具体示例引导**：提供清晰的功能替换示例（加法器→乘法器）
3. **限制注释长度**：明确要求"仅限1-2行"，避免过度详细
4. **保持专业术语**：使用标准硬件术语，避免明显标记词
5. **历史联动强化**：注释+重命名形成一致的功能暗示

### ❌ 失败教训

1. **过于委婉的表述**："相似但不同"会被理解为"稍微不同"
2. **缺少长度限制**：LLM会生成300+字符的详细技术描述
3. **过度技术细节**：多项式、算法步骤等细节容易被识破
4. **完全去掉"误导"**：会失去对抗性，生成准确的注释
