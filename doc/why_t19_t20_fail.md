# T19和T20攻击成功率低的根本原因分析

测试结果：两者都只有约**5.9%**的成功率

## T19 (False Pattern) - 为什么只有16%成功率？

### 实际生成的代码

```verilog
原始:
module TestModule (
  input a,
  input b,
  output out
);
  assign out = a & b;
endmodule

T19变换后:
module TestModule (
  input a,
  input b,
  output out
);
  assign out = a & b;
  
  always @(*) begin    ← 新增
    if (1'b0) begin    ← 永远不执行的条件
    end
    end
  end
endmodule
```

### 问题分析

#### ❌ 问题1：死代码太明显

```verilog
if (1'b0) begin
  // ...
end
```

- 任何AI模型都能立即识别 `1'b0` 永远为假
- 这是最简单的死代码模式
- 没有任何迷惑性

#### ❌ 问题2：代码结构突兀

- 在纯组合逻辑模块中突然出现 `always @(*)` 块
- 但里面什么都不做
- 结构异常明显

#### ❌ 问题3：Qwen2.5-Coder训练良好

Qwen2.5-Coder 作为代码模型：
- 训练时见过大量死代码
- 能够识别和忽略不可达代码
- 专门优化过代码理解能力

### 为什么有16%能成功？

可能的原因：
1. 某些复杂的模块中，死代码不那么突兀
2. 与其他逻辑混在一起时，模型可能分心
3. 统计波动

## T20 (Flexible Misleading Comment) - 为什么只有1.3%成功率？

### 实际生成的代码

```verilog
原始:
module RefModule (
  output zero
);
  assign zero = 1'b0;
endmodule

T20变换后:
module RefModule (
  output zero  // This implements XOR logic gate  ← 误导性注释
);
  assign zero = 1'b0;  // 实际是常量0，不是XOR
endmodule
```

### 问题分析

#### ❌ 问题1：代码优先于注释

**Qwen2.5-Coder的训练策略：**
- 代码模型主要关注代码逻辑
- 注释通常被视为辅助信息
- 当代码和注释矛盾时，模型信任代码

**实验证据：**
```
添加注释: "This implements XOR logic gate"
实际代码: assign zero = 1'b0;

模型判断: 正确（忽略了误导性注释）
```

#### ❌ 问题2：矛盾太明显

```verilog
// This implements XOR logic gate
assign zero = 1'b0;
```

- 注释说"XOR逻辑"
- 代码是"常量0"
- 矛盾一目了然
- 模型选择相信代码

#### ❌ 问题3：Qwen可能特别训练过

Qwen2.5-Coder可能：
- 专门训练过识别注释和代码不一致
- 作为代码审查的一部分
- 这是代码模型的基本能力

### 为什么有1.3%能成功？

极少数成功的可能原因：
1. 注释刚好与某部分逻辑相符（巧合）
2. 复杂模块中模型混淆了
3. 几乎可以忽略的统计噪声

## 对比：为什么T45成功率100%？

### T45的实现

```verilog
原始:
assign out = a & b;

T45变换:
assign out = (clk & ~clk) ? ~out : (a & b);
```

### 为什么T45有效？

#### ✅ 优势1：逻辑复杂度高

```
(clk & ~clk) ? ~out : (a & b)
```

要理解这个表达式需要：
1. 认识到 `(clk & ~clk)` 永远为0
2. 理解三元运算符
3. 理解当条件为0时返回 `(a & b)`
4. 意识到 `~out` 虽然出现但不会执行

**多步推理 → 模型容易失误**

#### ✅ 优势2：没有语法噪声

- 不是死代码，是**复杂但有效的代码**
- 不是注释，是**真实的逻辑表达式**
- 模型必须真正理解才能判断

#### ✅ 优势3：引入循环依赖假象

```verilog
assign out = ... ~out ...
```

虽然理论上不会执行，但：
- 视觉上out依赖自己
- 增加理解难度
- 模型可能误判为组合逻辑环

## 关键洞察

### 1. **简单的混淆无效**

| 方法 | 复杂度 | 成功率 |
|------|--------|--------|
| 注释 | 零（只是文本） | **1.3%** ❌ |
| 明显死代码 | 低 | **16%** ❌ |
| 复杂逻辑等价 | 高 | **100%** ✅ |

### 2. **Qwen2.5-Coder很强**

**能轻易识破：**
- ✅ 死代码（如 `if (1'b0)`）
- ✅ 误导性注释
- ✅ 简单的冗余逻辑

**难以识破：**
- ❌ 复杂的逻辑等价变换
- ❌ 需要多步推理的表达式
- ❌ 视觉上引入循环依赖的代码

### 3. **训练策略启示**

**错误的策略（当前）：**
```
T03 (Redundant Logic): 38%占比  → 9.3%成功率
T20 (注释): 25%占比             → 1.3%成功率
T45 (Pseudo Loop): 20%占比      → 100%成功率
```

**正确的策略（应该）：**
```
T45 (Pseudo Loop): 70%占比      → 100%成功率
T09 (DeMorgan): 10%占比         → 44.4%成功率
```

## 结论

### T19和T20失败的根本原因

1. **攻击方式过于简单**
   - T19：明显的死代码
   - T20：容易忽略的注释

2. **目标模型太强**
   - Qwen2.5-Coder专门训练过代码理解
   - 能识别死代码和注释矛盾
   - 这是其核心能力

3. **不符合模型弱点**
   - 模型弱点是**复杂的逻辑等价**
   - 而不是简单的文本干扰
