## 🎯 核心发现

### 发现1: Testbench模块名匹配问题

**原因**:
- Verilog数据集中，模块被重命名为`RefModule`
- Testbench期望原始模块名（如`mpc_controller`）
- `design_for_testbench`函数负责改回原始名字
- 但调用时缺少testbench参数，无法提取期望的模块名

**验证**:
```python
# 修复前
result = design_for_testbench(original, transformed)
# 模块名仍是RefModule → iverilog编译失败

# 修复后  
result = design_for_testbench(original, transformed, testbench)
# 模块名改为mpc_controller → iverilog编译成功 → testbench通过
```

### 发现2: Legacy的真实数据

**Legacy T19 (17,483个样本)**:
- Testbench通过: 43个 (0.3%)  ← GT攻击成功率
- LLM欺骗: 14,945个 (90.5%) ← 在原始正确样本中的欺骗率
- 真正的攻击成功: 很少（需要testbench通过 + Judge被欺骗）

**误解澄清**:
- ❌ **不是**: 90.5%的样本testbench通过
- ✅ **是**: 90.5%的原始正确样本中，LLM被死代码欺骗

### 发现3: T19的正确实现

**最终正确的T19策略**:
```verilog
// 在module内添加新的always块
always @(*) begin
  if (1'b0) begin
    // LLM生成的死代码（可以对真实信号赋值）
    signal <= value;
  end
end
```

**关键点**:
- ✅ 添加新的always块（不是在现有块内注入）
- ✅ 使用`if (1'b0)`包裹（或其他矛盾条件）
- ✅ 可以对真实信号赋值（因为代码不可达）
- ✅ 即使有多驱动，仿真器会优化掉死代码
