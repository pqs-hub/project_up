# 攻击数据集生成脚本使用说明

## 📖 概述

`pipeline/6_generate_attack_dataset.py` 是一个用于生成Verilog代码混淆攻击训练数据集的脚本。它通过遍历所有攻击规则，对启用CoT的判断模型进行攻击，只保留攻击成功的样本用于训练。

## 🎯 功能特性

### 核心功能

1. **遍历所有攻击规则**
   - T07: assign重排序
   - T19: 死代码注入
   - T20: 误导性注释
   - T32: 位宽算术混淆
   - T34: 内部信号重命名
   - T45: 常量替换
   - T47: 数据流分解
   - T48: 反拓扑排序

2. **LLM参数生成**
   - T20: 生成误导性注释内容
   - T34: 生成对抗性重命名映射
   - T19: 生成定制化死代码

3. **质量保证**
   - ✅ Testbench验证（功能等价性）
   - ✅ CoT判断模型验证（攻击成功性）
   - ✅ 只保留双重验证通过的样本

4. **输出格式**
   - JSONL格式，每行一个成功样本
   - 包含完整的上下文信息

## 🚀 快速开始

### 基础用法

```bash
# 生成数据集（不使用LLM参数生成）
python pipeline/6_generate_attack_dataset.py \
    --eval-file data/verilog_eval_correct_only.json \
    --output data/attack_dataset_$(date +%Y%m%d).jsonl \
    --max-samples 50 \
    --use-cot \
    --verbose
```

### 启用LLM参数生成

```bash
# 使用LLM生成更多样化的攻击参数
python pipeline/6_generate_attack_dataset.py \
    --eval-file data/verilog_eval_correct_only.json \
    --output data/attack_dataset_llm_$(date +%Y%m%d).jsonl \
    --max-samples 100 \
    --use-cot \
    --enable-llm-params \
    --attack-base-url http://localhost:8002/v1 \
    --attack-model "obfuscation_merged" \
    --verbose
```

### 指定规则子集

```bash
# 只测试特定规则
python pipeline/6_generate_attack_dataset.py \
    --eval-file data/verilog_eval_correct_only.json \
    --output data/attack_dataset_subset.jsonl \
    --max-samples 50 \
    --rules T19,T20,T34 \
    --use-cot \
    --verbose
```

## 📋 参数说明

### 必需参数

| 参数 | 说明 | 示例 |
|------|------|------|
| `--eval-file` | 输入的评估数据文件 | `data/verilog_eval_correct_only.json` |
| `--output` | 输出的JSONL文件路径 | `data/attack_dataset.jsonl` |

### 可选参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `--max-samples` | 最多处理多少个样本 | 全部 |
| `--rules` | 要测试的规则（逗号分隔） | 全部规则 |
| `--use-cot` | 启用CoT判断模式 | True |
| `--enable-llm-params` | 启用LLM参数生成 | False |
| `--attack-base-url` | 攻击模型API地址 | - |
| `--attack-model` | 攻击模型名称 | - |
| `--verbose` | 详细日志 | False |

## 📊 输出格式

### JSONL样本格式

每一行是一个JSON对象，包含：

```json
{
  "task_id": "Prob001_zero",
  "prompt": "实现一个输出恒为0的模块...",
  "original_code": "module zero(output y); assign y = 0; endmodule",
  "transformed_code": "module zero(output y); // 错误的注释\nassign y = 0; endmodule",
  "attack_rule": "T20",
  "attack_params": {"comment_text": "这是一个误导性注释"},
  "testbench_passed": true,
  "judge_fooled": true,
  "judge_confidence": 0.85,
  "judge_cot_original": "接口检查：输出y，符合要求。\n逻辑检查：assign y = 0，正确实现。\nFINAL_ANSWER: yes",
  "judge_cot_transformed": "接口检查：输出y，符合要求。\n注释说明：此注释有误导性。\n逻辑检查：代码与注释不符，可能有错误。\nFINAL_ANSWER: no",
  "judge_original_answer": "yes",
  "judge_transformed_answer": "no",
  "judge_original_correct": true
}
```

### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `task_id` | str | 任务标识符 |
| `prompt` | str | 功能规范描述 |
| `original_code` | str | 原始正确代码 |
| `transformed_code` | str | 攻击后的代码 |
| `attack_rule` | str | 使用的攻击规则ID |
| `attack_params` | dict | 攻击规则的参数 |
| `status` | str | 样本状态：success/attack_failed/testbench_failed/judge_error/no_change/exception |
| `failure_reason` | str | 失败原因（仅失败样本有此字段） |
| `testbench_passed` | bool | testbench是否通过 |
| `judge_fooled` | bool | 判断模型是否被欺骗 |
| `judge_confidence` | float | 判断模型的置信度 |
| `judge_cot_original` | str | 判断原始代码的CoT推理过程 |
| `judge_cot_transformed` | str | 判断变换后代码的CoT推理过程 |
| `judge_original_answer` | str | 原始代码的最终判断 (yes/no) |
| `judge_transformed_answer` | str | 变换后代码的最终判断 (yes/no) |
| `judge_original_correct` | bool | 原始代码是否被判为正确 |

### status状态说明

| 状态值 | 说明 |
|--------|------|
| `success` | 攻击成功（testbench通过且判断模型被欺骗） |
| `attack_failed` | 攻击失败（testbench通过但判断模型仍判对） |
| `testbench_failed` | testbench未通过（功能不等价） |
| `testbench_error` | testbench运行错误 |
| `judge_error` | 判断模型调用失败 |
| `no_change` | 变换未产生变化 |
| `exception` | 处理过程中发生异常 |

### 分析CoT差异

```python
# 统计CoT中的关键词
def analyze_cot_diff(cot_original, cot_transformed):
    keywords = ['错误', '不正确', '矛盾', '误导', '注释']
    
    for keyword in keywords:
        orig_count = cot_original.lower().count(keyword)
        trans_count = cot_transformed.lower().count(keyword)
        
        if trans_count > orig_count:
            print(f"'{keyword}' 在变换后增加: {orig_count} → {trans_count}")

# 使用
with open('data/attack_dataset.jsonl') as f:
    for line in f:
        sample = json.loads(line)
        analyze_cot_diff(
            sample['judge_cot_original'],
            sample['judge_cot_transformed']
        )
```

## ⚠️ 注意事项

### 1. 性能考虑

- **时间成本**: 每个样本×每个规则×每个参数组合 ≈ 5-10秒
- **建议**: 使用`--max-samples`限制规模，分批运行

### 2. 资源需求

- **GPU显存**: CoT模式需要更多显存（约8GB）
- **CPU**: Testbench运行需要CPU资源
- **网络**: LLM参数生成需要访问API

### 3. 数据质量

- 确保`--use-cot`启用，提高判断准确性
- 定期检查输出样本的质量
- 过滤重复样本

### 4. 错误处理

脚本会自动跳过：
- 原始代码判断错误的样本
- Testbench未通过的变换
- 攻击失败的样本

## 📞 问题排查

### Q: LLM参数生成失败？
**A**: 检查：
1. `--attack-base-url`和`--attack-model`是否正确
2. 攻击模型服务是否运行
3. 查看详细日志（`--verbose`）

### Q: 成功样本很少？
**A**: 可能原因：
1. CoT判断模型太强，难以欺骗
2. Testbench要求太严格
3. 规则参数组合不够多样化

解决方案：
- 增加`--max-samples`
- 启用`--enable-llm-params`
- 检查规则配置

### Q: 运行速度太慢？
**A**: 优化建议：
1. 减少`--max-samples`
2. 限制`--rules`范围
3. 分批并行运行
