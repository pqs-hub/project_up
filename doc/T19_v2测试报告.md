### 新发现的问题

#### 问题1：阻塞/非阻塞赋值混用

**示例** (q000000):
```verilog
always @(prev_control or disturbance) begin
    if (disturbance) begin
        control_signal = ...;  // ← 原始：阻塞赋值 (=)
    end
    
    // 死代码
    if (prev_control && !prev_control) begin
        control_signal <= ...;  // ← LLM生成：非阻塞赋值 (<=)
    end
end
```

**影响**: 在同一always块中混用`=`和`<=`可能导致：
1. Testbench检测到赋值方式变化
2. 仿真器认为功能不等价
3. 综合工具报警告

#### 问题2：LLM生成代码包含禁用关键字

**现象**: 3/9样本的LLM生成代码包含`always`关键字被过滤

**原因**: LLM有时会生成完整的always块结构：
```verilog
always @(posedge clk) begin
    // LLM生成的代码
end
```

**当前处理**: 直接跳过，导致`no_change`

## 需要解决的问题

### 1. 赋值方式一致性

**方案A**: 在Prompt中明确要求LLM使用与原代码相同的赋值方式
```
如果原始代码使用阻塞赋值(=)，你也必须使用阻塞赋值(=)
如果原始代码使用非阻塞赋值(<=)，你也必须使用非阻塞赋值(<=)
```

**方案B**: 后处理LLM输出，统一赋值方式
```python
# 检测原始代码的赋值方式
if '=' in original_always and '<=' not in original_always:
    assignment_style = '='
else:
    assignment_style = '<='

# 替换LLM生成代码中的赋值方式
llm_code = llm_code.replace('<=', assignment_style)
```

### 2. LLM生成always块的处理

**方案A**: 更智能的过滤
```python
# 提取always块内部的代码
if 'always @' in llm_code:
    # 提取begin...end之间的内容
    match = re.search(r'always\s*@.*?begin\s*(.*?)\s*end', llm_code, re.DOTALL)
    if match:
        llm_code = match.group(1)
```

**方案B**: 在Prompt中更严格的限制
```
严禁输出以下关键字：
- always
- initial  
- module
- endmodule
- begin (除非在if/case语句中)
- end (除非在if/case语句中)

只输出语句本体，例如：
✅ if (enable) signal = value;
✅ signal = value;
❌ always @(*) begin signal = value; end
```
