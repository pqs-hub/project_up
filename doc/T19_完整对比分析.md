## 关键洞察

### 洞察1: "No"是LLM真实输出 ✅

**证据**:
1. exhaustive中2个样本: `{"custom_dead_stmts": "No"}`
2. testbench_passed中2个样本: `{"custom_dead_stmts": "No"}`
3. 总共4个样本都包含相同的"No"输出

**结论**: LLM确实生成了字符串"No"，这不是解析错误或默认值。

### 洞察2: LLM生成成功率极低 ❌

**exhaustive.jsonl估算**:
```
假设遍历了 200个task × 15个规则 × 10个位置 = 30,000 次尝试
T19成功样本: 3个
T19有custom_dead_stmts: 2个
LLM参数生成成功率: 2/3000 ≈ 0.067% (假设T19占1/10)
```

**testbench_passed.jsonl实测**:
```
T19样本: 13个
有custom_dead_stmts: 2个
LLM参数生成成功率: 2/13 = 15.4%
```

**差异原因**: 
- exhaustive只保存攻击成功的 → 分母被严重筛选
- testbench_passed保存所有testbench通过的 → 分母更接近真实

**真实成功率估算**: **0.067% - 15.4%** 之间

### 洞察3: 空死代码竟然能欺骗Judge！ 🤔

**案例**: q000003

```verilog
always @(*) begin
  if (1'b0) begin
  end
end
```

**结果**:
- ✅ Judge fooled: True
- 置信度: 0.77 (中等)

**可能原因**:
1. **任务复杂度低** → Judge本身置信度就不高
2. **死代码位置关键** → 插入在敏感位置干扰了Judge推理
3. **Judge模型缺陷** → 对空死代码块的识别能力弱

### 洞察4: "No"语句也能欺骗Judge！ 🎯

**案例**: q010494, q015143

```verilog
always @(*) begin
  if ((2'b00 & 2'b11) == 2'b11) begin  // Impossible condition
    No
  end
end
```

**结果**:
- q010494: Judge置信度 0.65 (低)
- q015143: Judge置信度 0.95 (高！)

**可能原因**:
1. **Judge不理解"No"** → 可能认为是未定义信号或宏
2. **矛盾条件有效** → `(2'b00 & 2'b11) == 2'b11` 本身就有迷惑性
3. **运气成分** → q015143的高置信度可能是偶然

## 根本原因总结

### 问题1: LLM理解错误 ⚠️

**Prompt要求**:
```
Generate misleading dead code...
Output Verilog statements directly, no JSON
```

**LLM可能的理解**:
- "Generate no code" → 输出 "No"
- "No operation" → 简写为 "No"
- "Nothing to add" → 输出 "No"

**证据**: 4个样本都输出相同的"No"

### 问题2: 解析验证宽松 ❌

**当前逻辑** (6_generate_attack_dataset.py:360):
```python
return result if result else None
```

**问题**: 只要非空就返回，不验证是否合法Verilog

**后果**: "No"被接受为有效参数

### 问题3: API失败率高 📉

**testbench_passed数据**:
- 84.6%的调用没有返回custom_dead_stmts
- 可能原因: 超时、限流、返回空内容

**exhaustive数据**:
- 只有33.3%有custom_dead_stmts（但只统计成功样本）
- 实际失败率可能更高

### 问题4: 缺少语法验证 🔍

**现状**: 
- LLM输出 → 直接使用 → Iverilog编译
- 没有中间验证步骤

**后果**:
- "No"这样的无效代码直接插入
- 编译时可能报错（但某些情况下Iverilog容忍）
