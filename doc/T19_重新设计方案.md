# T19 重新设计方案

## 问题分析

### 当前T19的问题
1. **多驱动冲突**: 添加新的`always`块导致同一信号被多个块驱动
2. **Testbench失败率**: 100%失败（7/7样本）
3. **Verilog语法违规**: 即使死代码不执行，仍违反Verilog单驱动规则

### 失败示例
```verilog
// 原始代码
always @(*) begin
  signal = value1;
end

// T19添加的死代码（违反单驱动规则！）
always @(*) begin
  if (1'b0) begin  // 即使永不执行
    signal = value2;  // ❌ 多驱动冲突！
  end
end
```

## 重新设计方案

### 方案1：在现有always块内部注入（推荐）✅

**核心思路**: 不添加新的always块，而是在现有always块内部注入死代码

#### 实现策略

1. **检测现有always块**
   - 找到所有`always @(...) begin ... end`块
   - 优先选择最复杂的always块（行数最多）

2. **在always块内部注入**
   ```verilog
   // 原始
   always @(*) begin
     if (condition) begin
       signal = value1;
     end else begin
       signal = value2;
     end
   end
   
   // 修改后
   always @(*) begin
     if (condition) begin
       signal = value1;
     end else begin
       signal = value2;
     end
     
     // 注入的死代码（在同一个always块内）
     if (1'b0) begin  // 或矛盾条件
       // LLM生成的误导性代码
       if (enable && !enable) signal = value3;
     end
   end
   ```

3. **注入位置选择**
   - 优先级1：在最后一个条件分支后（if-else链的末尾）
   - 优先级2：在always块的末尾（end之前）
   - 优先级3：在某个begin-end块的末尾

#### 优点
- ✅ 不违反单驱动规则
- ✅ 死代码与真实逻辑混在一起，更隐蔽
- ✅ Testbench不会报语法错误
- ✅ LLM生成的代码更有迷惑性（看起来是逻辑的一部分）

#### 缺点
- ⚠️ 需要解析always块结构（稍复杂）
- ⚠️ 对纯assign语句的模块无效

### 方案2：只对纯组合逻辑模块注入

**核心思路**: 只在没有procedural赋值的模块中添加always块

#### 实现策略

1. **检测模块类型**
   ```python
   has_always = 'always @' in code
   has_initial = 'initial' in code
   
   if has_always or has_initial:
       # 跳过，不注入
       return code
   ```

2. **只对纯assign模块注入**
   ```verilog
   // 原始（纯组合逻辑）
   module RefModule(
     input a, b,
     output c
   );
     assign c = a & b;
   endmodule
   
   // 注入后
   module RefModule(
     input a, b,
     output c
   );
     assign c = a & b;
     
     // 添加always块（不冲突，因为原来没有）
     always @(*) begin
       if (1'b0) begin
         // 死代码
       end
     end
   endmodule
   ```

#### 优点
- ✅ 简单，易实现
- ✅ 避免多驱动冲突

#### 缺点
- ❌ 只适用于纯组合逻辑模块（覆盖率低）
- ❌ 对时序逻辑模块无效（大部分RTL都有always块）

### 方案3：使用注释形式的死代码

**核心思路**: 将死代码放在注释中，避免语法冲突

#### 实现策略
```verilog
always @(*) begin
  signal = value1;
  
  /* Dead code alternative implementation
  if (enable && !enable) begin
    signal = value2;  // Alternative logic (disabled)
  end
  */
end
```

#### 优点
- ✅ 完全避免语法冲突
- ✅ 对所有模块都适用

#### 缺点
- ❌ 不是真正的代码，可能被优化工具识别
- ❌ 迷惑性较弱

## 推荐方案：方案1（在现有always块内注入）

### 实现细节

```python
def ast_false_pattern_v2(
    code: str,
    target_token: Optional[int] = None,
    target_line: Optional[int] = None,
    target_signal: Optional[str] = None,
    custom_dead_stmts: Optional[str] = None,
) -> str:
    """T19 v2: 在现有always块内部注入死代码"""
    
    vs = analyze(code)
    
    # 1. 找到所有always块
    always_blocks = find_always_blocks(code)

        # 没有always块，回退到方案2（只对纯组合逻辑）

        # 在endmodule前添加always块（安全，因为没有现有的）

    # 2. 选择要注入的always块（选择最复杂的）
    target_block = select_target_always_block(always_blocks, target_token, target_line)

```

### 关键函数

#### 1. 找到always块
```python
def find_always_blocks(code: str) -> List[Dict]:
    """找到所有always块的位置和范围"""
    blocks = []
    
    # 正则匹配always块
    pattern = r'always\s*@\s*\([^)]+\)\s*begin'
    
    for match in re.finditer(pattern, code):
        start = match.start()
        # 找到对应的end
        end = find_matching_end(code, match.end())
        
        blocks.append({
            'start': start,
            'end': end,
            'header': match.group(),
            'body': code[match.end():end],
        })
    
    return blocks
```

#### 2. 在always块内注入
```python
def inject_inside_always_block(code: str, block: Dict, custom_stmts: str) -> str:
    """在always块内部注入死代码"""
    
    # 构建死代码（带矛盾条件）
    dead_code = f"""
    // Dead code path (unreachable)
    if (1'b0) begin
{indent_code(custom_stmts, 6)}
    end"""
    
    # 在always块的end之前插入
    inject_pos = block['end'] - 3  # 在"end"之前
    
    return code[:inject_pos] + dead_code + "\n  " + code[inject_pos:]
```
