**当前评估数据缺少`original_passed`字段（全部为None），无法计算真正的ASR！**

原因：评估时只运行了`adversarial`模式，没有评估原始代码的LLM判断。

从功能破坏分析可以看出，很多"破坏"样本的`adversarial_code == 'NO'`，说明规则根本没有应用。

- **LLM原始准确率：0.0%**（因为没有评估）

**这说明在功能等价的情况下，LLM对对抗代码的判断是正确的！**

## 🚨 问题规则

### T47 - 严重的功能破坏

- **功能等价率：5.0%**（最低）
- **功能破坏：740个样本**
- **规则应用率：17.2%**（很低）

**分析**：
- 大部分样本功能被破坏
- 规则应用率也很低
- **需要重点修复**

### T31 - 中等功能破坏

- **功能等价率：42.7%**
- **功能破坏：3,658个样本**
- **规则应用率：59.6%**

**分析**：
- 超过一半样本功能被破坏
- **需要修复**

### T10, T18, T48, T12, T09 - 轻微功能破坏

- **功能等价率：63-80%**
- **功能破坏：172-494个样本**

**分析**：
- 有一定比例的功能破坏
- 可以考虑优化

## 💡 功能破坏的真实原因

### 规则未应用占比

从抽样分析（前5个功能破坏样本）：

| 规则 | 功能破坏数 | 其中code=NO | 占比 |
|------|-----------|------------|------|
| T09 | 172 | 5/5 | **100%** |
| T07b | 5 | 5/5 | **100%** |
| T34 | 253 | 5/5 | **100%** |
| T45 | 61 | 5/5 | **100%** |
| T47 | 740 | 5/5 | **100%** |
| T12 | 304 | 4/5 | **80%** |
| T18 | 225 | 4/5 | **80%** |
| T19 | 43 | 4/5 | **80%** |
| T10 | 300 | 3/5 | **60%** |
| T48 | 494 | 3/5 | **60%** |

**结论**：
- **大部分"功能破坏"是因为规则根本没有应用！**
- 当`adversarial_code == 'NO'`时，`adversarial_truth`被标记为False
- 这不是真正的功能错误，而是规则不适用

## 🎯 LLM判断分析

### 在功能等价且规则应用的样本上

**所有规则的LLM对抗准确率都是100%！**

这说明：
- 在功能等价的情况下
- LLM对对抗代码的判断是**正确的**
- **没有攻击成功的样本**

### 为什么会这样？

可能的原因：
1. **评估模型太强**：能够正确理解对抗变换
2. **对抗强度不够**：变换不足以欺骗LLM
3. **评估方式问题**：prompt或评估方法不合适

---

## 📌 结论

### 1. 无法计算真正的ASR

因为缺少`original_passed`数据，无法计算：
```
ASR = (original_passed==True AND adversarial_passed==False) / (original_passed==True)
```

### 2. 功能等价性总体良好

- **91.6%的样本保持功能等价**
- 大部分规则工作正常

### 3. "功能破坏"主要是规则未应用

- 不是真正的功能错误
- 是规则不适用于某些代码模式

### 4. LLM对对抗代码判断全部正确

- 在功能等价的样本上
- LLM准确率100%
- **没有成功的对抗攻击**
