# 多规则冲突问题及解决方案

## ⚠️ 问题分析

### 数据统计
```
独特任务数: 16,141
总攻击成功样本: 44,246
平均每个任务: 2.74 个成功规则

✅ 只被1个规则攻击: 2,801 个任务 (17.4%)
⚠️  被多个规则攻击: 13,340 个任务 (82.6%)

冲突样本占比: 93.7%
```

### 问题描述

**82.6%的任务被多个规则攻击**，导致：

#### 同一Input有多个Output
```
Input (任务 q013332):
  功能规范: xxx
  原始代码: yyy

Output 1: {"attack_name": "redundant_logic", ...}       // T03
Output 2: {"attack_name": "demorgan_and", ...}          // T09  
Output 3: {"attack_name": "false_pattern_injection",...} // T19
... (共11个不同答案)
```

#### 训练问题
1. **模型混乱**: 同一输入对应多个"正确"输出
2. **梯度冲突**: 不同batch的相同样本pull向不同方向
3. **预测不稳定**: 测试时无法确定应该选择哪个规则

---

## 💡 解决方案对比

### 方案1: 去重（每任务保留最佳规则）⭐ 推荐

#### 策略
- 每个任务只保留**ASR最高**的规则
- 确保每个input只有一个output
- 优先保留最有效的混淆

#### 实施
```bash
python deduplicate_sft_dataset.py
```

#### 结果
```
原始样本: 44,246
去重后: 16,126 (减少63.6%)

规则分布:
  T45 (ASR=99.1%): 6,193 (38.4%) ← 占比最大
  T19 (ASR=90.8%): 9,218 (57.2%) ← 占比最大
  T20 (ASR=51.6%):   490 ( 3.0%)
  T34 (ASR=42.1%):   175 ( 1.1%)
  其他规则: < 0.1%
```

#### 优点
✅ **解决冲突**: 每个任务只有一个答案
✅ **训练稳定**: 无梯度冲突
✅ **聚焦高效**: 保留最有效的规则

#### 缺点
⚠️ **规则不平衡**: T45+T19占95.6%
⚠️ **规则多样性下降**: 低ASR规则样本极少
⚠️ **泛化能力**: 模型可能只学会T45和T19

---

### 方案2: 多任务学习（保留所有规则）

#### 策略
- 保留所有44,246个样本
- 训练时随机采样
- 让模型学习**多种可能**的混淆策略

#### 实施
```python
# 训练时使用随机采样
dataloader = DataLoader(
    dataset, 
    batch_size=32,
    shuffle=True,  # 随机打乱
    sampler=None   # 不做特殊处理
)
```

#### 优点
✅ **规则多样性**: 保留所有15个规则
✅ **数据量大**: 44,246个样本
✅ **真实反映**: 一个代码确实可以用多种方式混淆

#### 缺点
⚠️ **训练不稳定**: 同一样本可能被当作不同标签
⚠️ **收敛困难**: 梯度冲突
⚠️ **需要特殊处理**: 如重新设计loss函数

### 方案3: 分层采样（平衡去重）⭐ 推荐

#### 策略
- 每个任务最多保留**N个规则**（如N=3）
- 优先保留**不同ASR层级**的规则
- 平衡高ASR和规则多样性

#### 实施
```python
def balanced_dedup(samples_per_task, max_rules_per_task=3):
    """每个任务最多保留N个规则，按ASR分层选择"""
    
    # 定义ASR层级
    high_asr = ['T45', 'T19']           # ASR > 80%
    mid_asr = ['T30', 'T12', 'T20']     # 50% < ASR < 80%
    low_asr = ['T34', 'T09', 'T31', ...] # ASR < 50%
    
    selected = []
    
    for task, rules in samples_per_task.items():
        # 从每个层级选1个
        for tier in [high_asr, mid_asr, low_asr]:
            tier_rules = [r for r in rules if r in tier]
            if tier_rules and len(selected) < max_rules_per_task:
                # 选择该层级中ASR最高的
                best = max(tier_rules, key=lambda r: RULE_ASR[r])
                selected.append(best)
        
        # 如果还不够，补充高ASR规则
        while len(selected) < max_rules_per_task and len(rules) > len(selected):
            # ...
    
    return selected
```

#### 结果（估算）
```
原始: 44,246
平衡去重后: ~35,000 (保留率~79%)

规则分布（更均衡）:
  T45: ~5,000 (14%)
  T19: ~8,000 (23%)
  T20: ~4,000 (11%)
  T34: ~3,000 (9%)
  T12: ~3,000 (9%)

```

#### 优点
✅ **解决冲突**: 限制每任务的规则数
✅ **保留多样性**: 各规则都有足够样本
✅ **平衡ASR**: 高中低ASR规则都有

#### 缺点
⚠️ **实现复杂**: 需要设计采样策略
⚠️ **仍有冲突**: max_rules_per_task>1时仍有冲突

### 方案4: 改变任务定义（多标签分类）

#### 策略
- 不是选择"一个"规则
- 而是输出"所有适用"的规则
- 改为**多标签分类**任务

#### Output格式改变
```json
// 原格式（单规则）
{
  "attack_name": "redundant_logic"
}

// 新格式（多规则）
{
  "applicable_rules": [
    {"attack_name": "redundant_logic", "priority": 3},
    {"attack_name": "demorgan_and", "priority": 2},
    {"attack_name": "false_pattern_injection", "priority": 1}
  ]
}
```

#### 优点
✅ **符合现实**: 一个代码确实可用多规则
✅ **无冲突**: 所有规则都是"正确答案"
✅ **更灵活**: 可按priority选择

#### 缺点
⚠️ **改变任务**: 需要重新设计整个pipeline
⚠️ **难以评估**: 如何判断预测正确？
⚠️ **应用困难**: 实际攻击时如何选择规则？
