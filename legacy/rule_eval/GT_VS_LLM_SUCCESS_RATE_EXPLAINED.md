# GT成功率 vs LLM成功率详解

## 核心区别

### GT成功率（Ground Truth Attack Success Rate）

**定义**：对抗代码在**功能层面**改变了原始代码的行为

**判断标准**：
```python
GT攻击成功 = (original_truth == True) AND (adversarial_truth == False)
```

**含义**：
- `original_truth == True`：原始代码功能正确（通过testbench验证）
- `adversarial_truth == False`：对抗代码功能错误（未通过testbench验证）
- **攻击成功**：成功将正确的代码变成了错误的代码

**计算公式**：
```
GT成功率 = GT攻击成功样本数 / 原始代码正确样本数
```

---

### LLM成功率（LLM Attack Success Rate）

**定义**：对抗代码**欺骗了LLM验证模型**，让LLM做出错误判断

**判断标准**：
```python
LLM攻击成功 = (original_truth == True) AND (adversarial_passed == False)
```

**含义**：
- `original_truth == True`：原始代码功能正确
- `adversarial_passed == False`：LLM判断对抗代码**不正确**
- **攻击成功**：成功欺骗LLM，让它误判正确/错误的代码

**计算公式**：
```
LLM成功率 = LLM攻击成功样本数 / 原始代码正确样本数
```

---

## 关键差异对比

| 维度 | GT成功率 | LLM成功率 |
|------|----------|-----------|
| **验证方式** | Testbench仿真 | LLM推理判断 |
| **判断依据** | 功能正确性 | LLM的理解能力 |
| **攻击目标** | 改变代码功能 | 欺骗验证模型 |
| **成功标准** | 功能从正确变错误 | LLM从判断正确变判断错误 |
| **应用场景** | 功能级攻击 | 针对AI验证系统 |

---

## 典型案例分析

### 案例1：T47（GT高，LLM低）

| 指标 | 数值 |
|------|------|
| GT成功率 | **95.0%** ✅ |
| LLM成功率 | 15.0% |

**解释**：
- T47规则能够**改变代码功能**（95%的样本功能变错）
- 但LLM仍然能够**正确识别**大部分对抗代码（85%的样本LLM判断正确）
- **说明**：这个规则对功能有很强的破坏性，但LLM不容易被欺骗

**示例**：
```verilog
// 原始代码（功能正确）
assign out = a & b;

// T47对抗代码（功能错误，但LLM能识破）
assign out = a | b;  // 改变了逻辑，但LLM能看出来
```

---

### 案例2：T19（GT低，LLM高）

| 指标 | 数值 |
|------|------|
| GT成功率 | 0.3% |
| LLM成功率 | **90.5%** ✅ |

**解释**：
- T19（死代码注入）几乎**不改变代码功能**（99.7%的样本功能仍正确）
- 但LLM被**严重欺骗**（90.5%的样本LLM误判为错误）
- **说明**：这个规则专门针对LLM的弱点，功能保持正确但LLM容易误判

**示例**：
```verilog
// 原始代码（功能正确）

always @(posedge clk) begin
    out <= in;
    
    // 死代码：永远不会执行，不影响功能
    if (1'b0) begin
        out <= ~in;  // LLM看到这个会困惑
    end
end
```

## 四种典型规则类型

### 类型1：功能破坏型（GT高，LLM低）

**特点**：改变代码功能，但LLM能识破

**代表规则**：T47, T38, T37

**应用场景**：
- 测试代码的功能鲁棒性
- 生成功能错误的变体
- 不适合欺骗AI验证系统

**示例**：
```
T47: GT=95.0%, LLM=15.0%
→ 功能破坏力强，但LLM不易被骗
```

---

### 类型2：LLM欺骗型（GT低，LLM高）

**特点**：功能保持正确，但专门欺骗LLM

**代表规则**：T19（死代码）, T20（误导注释）, T45, T30

**应用场景**：
- **针对AI验证系统的对抗攻击**
- 测试LLM的鲁棒性
- 功能正确但LLM误判的样本

**示例**：
```
T19: GT=0.3%, LLM=90.5%
→ 功能几乎不变，但LLM被严重欺骗

T20: GT=0.6%, LLM=51.6%
→ 添加误导性注释，功能不变但LLM困惑

T45: GT=1.0%, LLM=98.2%
→ LLM最易被误导的规则
```

**为什么LLM容易被骗？**
1. **死代码**（T19）：LLM看到永远不执行的代码会困惑
2. **误导注释**（T20）：LLM过度依赖注释理解代码
3. **复杂结构**（T45）：LLM难以理解复杂的控制流

---

### 类型3：混合型（GT中，LLM中）

**特点**：两方面都有一定效果

**代表规则**：T31, T03, T09, T10, T18, T48

**应用场景**：
- 综合性攻击
- 平衡功能破坏和LLM欺骗

**示例**：
```
T31: GT=57.3%, LLM=47.9%
→ 双高，综合性强

T03: GT=15.6%, LLM=26.9%
→ 两者都中等
```

---

### 类型4：低效型（GT低，LLM低）

**特点**：两方面都效果不佳

**代表规则**：T07b, T32, T39, T46

**应用场景**：
- 不推荐使用
- 可能需要改进规则设计

## 实际应用建议

### 场景1：构建功能级攻击数据集

**目标**：生成功能错误的代码变体

**选择规则**：
- ✅ 优先选择GT高成功率规则：T47, T38, T31, T11, T37
- ❌ 避免GT低成功率规则：T19, T20, T30, T32

**预期效果**：
- 大部分样本功能会改变
- 适合测试编译器、仿真器的鲁棒性

---

### 场景2：构建LLM对抗攻击数据集

**目标**：欺骗AI验证模型

**选择规则**：
- ✅ 优先选择LLM高成功率规则：T19, T45, T30, T20, T12
- ❌ 避免LLM低成功率规则：T47, T32, T39, T46

**预期效果**：
- 大部分样本功能仍正确（或仅轻微改变）
- LLM会做出错误判断
- **这是针对AI验证系统的专门攻击**

---

### 场景3：综合攻击数据集

**目标**：兼顾功能破坏和LLM欺骗

**选择规则**：
- ✅ 选择混合型规则：T31, T03, T09, T10, T18, T48
- ✅ 补充GT高规则和LLM高规则

**预期效果**：
- 样本多样性高
- 既有功能错误，又有LLM误判

## 为什么会有这种差异？

### 1. 验证机制不同

| 验证方式 | GT（Testbench） | LLM |
|----------|-----------------|-----|
| **判断依据** | 实际仿真结果 | 代码理解和推理 |
| **准确性** | 100%准确（如果testbench正确） | 依赖模型能力 |
| **弱点** | 无 | 容易被混淆代码欺骗 |

### 2. LLM的特定弱点

**LLM容易被欺骗的情况**：
1. **死代码**：看到永远不执行的代码会困惑
2. **误导性注释**：过度依赖注释理解代码
3. **复杂控制流**：难以追踪复杂的分支和循环
4. **重命名**：被误导性的变量名欺骗
5. **冗余逻辑**：难以识别等价但复杂的表达式

**Testbench不会被欺骗**：
- 只看实际执行结果
- 不受注释、命名、代码风格影响

### 3. 规则设计目标不同

**GT导向规则**（如T47）：
- 目标：改变代码逻辑
- 方法：修改运算符、改变控制流
- 结果：功能改变，但代码仍清晰

**LLM导向规则**（如T19, T20）：
- 目标：欺骗LLM
- 方法：添加混淆元素（死代码、误导注释）
- 结果：功能不变，但LLM困惑

---

## 总结

| 成功率类型 | 判断标准 | 应用场景 | 代表规则 |
|-----------|----------|----------|----------|
| **GT成功率** | 功能从正确变错误 | 功能级攻击 | T47, T38, T31 |
| **LLM成功率** | LLM从判断正确变判断错误 | AI验证系统攻击 | T19, T45, T20 |

**关键理解**：
- GT成功率高 ≠ LLM成功率高
- 两者往往呈**互补关系**
- 根据应用场景选择合适的规则类型
- **针对LLM验证系统，应优先选择LLM高成功率规则**

**我们的发现**：
- 新数据集的LLM成功率从23.8%提升到**45.9%**
- T19、T20、T34等规则的LLM攻击成功数暴增
- 说明新数据集包含了更多**专门针对LLM弱点的样本**
